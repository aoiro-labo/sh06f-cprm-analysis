# SH-06F ワンセグ・フルセグ録画 CPRM 解析記録

**対象機器**: SHARP SH-06F（Android 4.4.2, docomo, 2014年発売）  
**記録媒体**: 32GB microSDHC / Panasonic BN-SDCMP3（CPRM対応リーダー）  
**実施期間**: 2025年12月〜2026年  
**状態**: ワンセグ＝画面キャプチャで救出済 ／ フルセグ＝未解決

---

## このリポジトリについて

SH-06F で録画したワンセグ・フルセグ映像を、**2026年時点で入手可能な正規手段が事実上存在しない**という状況で、
やむを得ず技術的に解析・救出するために行ったリバースエンジニアリングの記録です。

- 商用目的・海賊版流布の意図は一切ありません
- 録画した本人が自分のコンテンツを参照するための作業です
- CPRM の暗号方式・鍵構造の学術的理解を記録として残します
- 将来の ISDB-T/S/4K8K 規格がこの失敗を繰り返さないよう提言します

## 1. 動機と「やむを得ない」事情

### 1-1. 正規手段の消滅経緯

| 時期 | 出来事 |
|------|--------|
| 2006 | Panasonic SD-MobileImpact 発売（Windows XP 対応 CPRM 再生ソフト） |
| 2007 | SDHC 対応パッチ (L042→L052) 提供、ただし SDHC フル対応は後年版 |
| 2014/4 | Windows XP サポート終了。XP 向けソフトの動作保証がなくなる |
| 2018/3 | **Panasonic がセキュリティプログラム配布を完全終了**。新規インストール・再インストール不可に |
| 2019 | Windows 10 で OS レベルの CPRM サポートが完全撤去 |
| 2026/現在 | SD-MobileImpact が動作する唯一の環境は Windows XP SP3 (2014年 EOL) |

**つまり**: 2026年時点で、ユーザーが自分で録画した SD-Video(ワンセグ) コンテンツを正規に再生・救出する手段は
事実上存在しません。SH-06F 本体が壊れていたり手元になければ、動画は閲覧不可能です。

### 1-2. 試みた正規経路と失敗

| 試みた経路 | 結果 | 失敗理由 |
|------------|------|----------|
| SD-MobileImpact を現行 Windows 11 で起動 | ❌ 86000001 クラッシュ | `mirssom.dll` use-after-unload（OS の DLL ローダ仕様変化） |
| SDHC 対応パッチ (L052) への更新 | ❌ 詰み | 追加モジュール配布サーバが 2018 年にサ終 |
| Panasonic DIGA で再生・ダビング | ❌ 不可 | 「持ち出し番組の再生はできません」。録画機紐付けを検知 |
| WSL/Linux での直接読み出し | ❌ 鍵なし | CPRM 保護領域は AKE (認証鍵交換) なしでは読めない |
| StationTV アプリの解析 | ❌ 進まず | APK に native .so が入っておらず、復号ロジック本体が取得不可 |
| **XP SP3 VM + SD-MobileImpact + バイナリパッチ** | **✅ 成功** | 詳細は §4 / `doc/ワンセグ救出_再現手順.md` |

### 1-3. 法的・倫理的立場

**著作権法第30条（私的使用のための複製）**
録画した本人が、自分が視聴するために自分の録画データを複製・変換することは、同条の私的使用の範囲内と解釈します。

**著作権法第30条の4（情報解析）**
暗号方式・ファイル構造の解析は情報解析研究の範囲です。
得られた知見の学術的公表は正当な情報公開です。

**「やむを得ない事情」の要件**
- 正規再生ソフトの配布が終了し、入手・更新が不可能
- 正規ソフトが動作する OS が EOL（XP：2014年終了）
- 録画機（SH-06F）に依存した DRM で、機器廃棄後のアクセス手段がない
- 代替となる公式移行手段が提供されないまま製品・サービスが終了
- コンテンツは地上波無料放送の個人録画（商業価値なし）

## 2. 技術的成果サマリ

### 2-1. ファイル構造（確定）

```
SD_VIDEO/PRGxxx/MOV001.sb1 （ワンセグ、SD-Video 規格）
─────────────────────────────────────────
各 6144 B (0x1800) ユニットの繰り返し:
  ┌ 0x000–0x0BF (192B) 平文ユニットヘッダ
  │   0x000–0x003 : 00 00 00 00  (パディング)
  │   0x004–0x0BB : 188B TS-like パケット (PID=0x001F, 平文)
  │   0x060       : 08 00 1F 00  (副マーカー)
  │   0x09F       : 39           (固定フラグ)
  │   0x0A0–0x0BF : 32B ETK 候補領域 (4×8B、暗号化タイトル鍵関連)
  └ 0x0C0–0x17FF (5952B) 暗号化ペイロード (31×192B 暗号化 TS パケット)

暗号モード : C2-CBC (64bit ブロック, ユニット毎 IV リセット)
タイトル鍵 : 56bit (7B、第8B は 0x00 パディング)
鍵は録画ごとに異なる (18本すべて先頭暗号ブロックが相違)
```

### 2-2. 鍵階層（CPRM）

```
[デバイス鍵 (秘密・ライセンス品)]
    + [MKB (カードのシステム領域)] → Km (メディア鍵)
Km + [Media ID (カード製造時読取専用)] → Kmu (メディア固有鍵)
Kmu で [ETK (暗号化タイトル鍵、保護領域)] を C2 復号 → TitleKey (56bit)
TitleKey で .sb1 ペイロードを C2-CBC 復号 → 平文 TS
```

**ここで詰む**: Media ID・MKB・ETK はいずれも SDカードの保護領域／読取専用領域にあり、
通常のファイルコピーには絶対に含まれません。CPRM 認証コマンド (AKE) 経由でのみ読めます。

### 2-3. C2 (Cryptomeria) 暗号の実装解析

- **ブロック長**: 64bit (8B)
- **ラウンド数**: 16 ラウンド
- **鍵長**: 56bit (7B、第8Bを0x00にパディングして C2 鍵スケジュールへ入力)
- **構造**: XOR Feistel。`L,R ← R, F(R,K_r) ⊕ L` を 16 回
- **ラウンド関数 F**: S-box → 線形混合 → ラウンド固有の混合演算 (⊕/+/- が ROUND_OPS で変化)
- **復号の注意点**: ラウンド鍵の逆順 (`K[15-r]`) に加え、**ROUND_OPS も逆順** (`ROUND_OPS[15-r]`) が必須
  - 誤り例: `K[15-r] + ROUND_OPS[r]` → 復号失敗 (これが当初の実装バグ)
  - 正解: `K[15-r] + ROUND_OPS[15-r]` → 正しい復号 (`test_c2_roundtrip2.py` で検証済み)
- **S-box**: `SDVM2TSPacketDecParser.dll` の実行時メモリから抽出。`doc/_migration_kit/c2_sbox.bin` に保存
  - 先頭 16B: `f7 3a cd 29 fc 11 17 fe 39 36 86 48 f3 e2 af aa`

### 2-4. MEI006H アーキテクチャ（今回最重要な発見）

SD-MobileImpact のメモリをいくら解析しても TitleKey が見つからない理由が判明しました。

**全ての CPRM 鍵演算は `C:\Windows\SysWOW64\mei006h.exe`（Windows サービス）の内部で完結**します。
SD-MobileImpact はコンテンツ再生中、復号済みデータだけを名前付きパイプ経由で受け取るため、
**ユーザーモードのプロセスメモリには TitleKey も Kmu も一切現れません**。

```
SD-MobileImpact.exe (ユーザープロセス)
    ↓ 名前付きパイプ \\.\pipe\mei006h
MEI006H.exe (Windows サービス, SysWOW64)
    → DeviceKey + MKB → Km
    → Km + MediaID → Kmu
    → Kmu + ETK → TitleKey
    → TitleKey + C2-CBC で .sb1 を復号
    → 平文コンテンツをパイプ経由で返す
```

- `mei006h.exe` のパイプに Authenticated Users が GRGWGX でアクセス可能なセキュリティ記述子
- DLL ロードパターン: `%s\%s\%s\%016I64X%016I64X.DLL`
- 内部で `alterstar` 文字列、複数 GUID を参照
- **管理者権限なしではダンプ不可** (AccessDenied)

### 2-5. メモリスキャン結果（証拠）

修正後 C2 実装 (`ROUND_OPS[15-r]`) を使用し、全メモリダンプをスキャン:

| ダンプ | スキャン種別 | ヒット数 | 備考 |
|--------|-------------|---------|------|
| dump_10 | TitleKey (R0+L0 両モード) | **0** | SD-MobileImpact 起動中 |
| dump_11 | TitleKey (R0+L0 両モード) | **0** | コンテンツ再生開始直後 |
| dump_11 | Kmu → TitleKey 二段階 | **0** | ETK 4候補 × 3形式 |
| dump_11 | alt-oracle (逆方向) | **0** | C2 暗号化方向の誤り確認用 |
| dump_12 | TitleKey (R0+L0 両モード) | **0** | |

→ SD-MobileImpact ユーザーモードメモリに鍵材料が存在しないことを多角的に確認。
  MEI006H サービス内への分離アーキテクチャが裏付けられた。

## 3. 証拠データ

### 3-1. .sb1 ファイルの先頭暗号ブロック（18 本全て相違）

`SD_VIDEO/PRG011/MOV001.sb1` の場合:
```
ETK @ 0x0A0: 03cd10ab81550359 f5fa873f8def2a07 1993d56c85a246d6 243b745f9c820164
CT[0] @ 0xC0: 7f782f4a9fed30ae  (最初の暗号化TS 8バイト)
CT[1] @ 0x180: 3d83dff73bc3efce
CT[2] @ 0x240: 24b4b44c2c6dfd3e
```

ETK は 6144B ごとに同一値が繰り返し現れる（全74ユニット）: ファイル構造の正確な証拠。

### 3-2. C2 S-box（最初の 16 バイト）

```
f7 3a cd 29 fc 11 17 fe 39 36 86 48 f3 e2 af aa ...
```
（`doc/_migration_kit/c2_sbox.bin` に 256バイト全体を保存）

### 3-3. C2 ラウンドトリップ検証（`scripts/test_c2_roundtrip2.py`）

```
dec_v2 [rev_K + rev_OPS]:  PT=47010203-04050607 → CT=*** → dec=47010203-04050607  [MATCH]
dec_v1 [rev_K + fwd_OPS]:  PT=47010203-04050607 → CT=*** → dec=XXXXXXXX-XXXXXXXX  [FAIL]
```
16ラウンド Feistel の復号は鍵とラウンド演算の両方を逆順にする必要があることを実証。

## 4. ワンセグ救出の実現方法（概要）

詳細: [`doc/ワンセグ救出_再現手順.md`](doc/ワンセグ救出_再現手順.md)

1. **Windows XP SP3 32bit VM**（VMware Workstation Player）を構築
2. SD-MobileImpact **v1.0.61220.1000 (L032)** を正規 CD-ROM ISO からインストール  
   ⚠️ L052 (Vista 対応版) に更新しない（追加 DL サーバが 2018 年にサ終で詰む）
3. 86000012 (SDHC を「ファイルシステムが異なります」と拒否) を 1バイトパッチで回避:
   ```
   ファイルオフセット 0x429E9: 75 → EB  (jne → jmp)
   ファイルオフセット 0x78EBB: 75 → EB  (jne → jmp)
   ```
4. ゲスト OS のハードウェアアクセラレータを「なし」に設定（黒画面回避）
5. BN-SDCMP3 を USB 2.0 経由で VM にパススルー
6. 再生 → OBS で画面・音声キャプチャ

**フルセグ (`PRIVATE/SHARP/FSEG/`)**: SHARP 独自形式・デバイス紐付きのため、現時点では未解決。
メタデータ（題名・局・日時）のみ SQLite DB から救出済み。
