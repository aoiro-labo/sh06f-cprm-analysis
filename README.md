# SH-06F ワンセグ・フルセグ録画 CPRM 解析記録

**録画機器**: SHARP SH-06F（Android 4.4.2 KitKat / docomo / 2014年発売）  
**記録媒体**: 32GB microSDHC（FAT32 / 空き約 24.6GB）  
**CPRMリーダー**: Panasonic BN-SDCMP3（CPRM対応 / Windows 認識名: BN-SDCM Series）  
**TVアプリ**: PIXELA StationTV v1.5-r30503（`jp.pixela.stationtv.localtuner.full.app`）  
**受信地点**: 名古屋市内 中部電力パワーグリッド MIRAI TOWER（旧名古屋テレビ塔）付近（2025年12月7日の録画分）  
**受信エリア**: 愛知・三重（名古屋局系）  
**解析実施**: 2025年12月〜2026年6月  
**ワンセグ状態**: 画面キャプチャによる映像救出済み  
**フルセグ状態**: 未解決（SHARP 独自形式、メタデータのみ復元済み）

---

## 概要

SH-06F で録画したワンセグ（SD-Video / CPRM 暗号化）およびフルセグ（SHARP 独自形式）の映像データを
個人的に保存・閲覧するため、暗号方式とソフトウェアスタックのリバースエンジニアリングを実施した記録である。

**問題の核心**: CPRM 対応の正規再生ソフト（SD-MobileImpact）は 2018 年 3 月に配布終了し、
動作する OS は Windows XP（2014 年 EOL）のみとなっている。
2026 年時点で、ユーザーが自分で録画した映像を正規手段で再生・保存する方法は事実上存在しない。

本リポジトリには解析スクリプト・技術ドキュメント・暗号構造の解析成果物・メタデータを収録する。
**映像本体（.sb1 / FSEG）およびメモリダンプは除外**している。

### 3-2. .sb1 ファイル構造（実測確定）

ファイルサイズは必ず 2048 の倍数（2KB セクタ単位）かつ 6144 の倍数（0x1800 = 6144B）。

```

1ユニット (6144B = 0x1800B):
 ┌──────────────────────────────────┐ offset +0x000
 │ 平文ユニットヘッダ (192B = 0xC0B) │
 │  +0x000: 00 00 00 00             │  パディング（4B）
 │  +0x004: 47 00 1F 1X 00 80 30 B4 │  TS-like マーカー（PID=0x001F）
 │           X = ユニット連続性カウンタ (0,1,2…)
 │  +0x060: 08 00 1F 00             │  副マーカー（固定）
 │  +0x09F: 39                      │  固定フラグ
 │  +0x0A0〜+0x0BF: ETK 候補 (32B)  │  暗号化タイトル鍵関連 (4×8B)
 └──────────────────────────────────┘
 ┌──────────────────────────────────┐ offset +0x0C0
 │ 暗号化ペイロード (5952B = 0x1740B)│  31 × 192B 暗号化 TS パケット
 │ 暗号モード: C2-CBC (8B ブロック)  │
```

## 4. CPRM 鍵階層と「詰む」理由

```
[デバイス鍵]                    → 録画機器内部に格納。外部取り出し不可。
    + [MKB]                    → SDカード システム領域。AKE なしでは読み出し不可。
    ↓ C2 復号
  メディア鍵 Km

Km + [Media ID]                → SDカード 製造時固有値（読取専用領域）。AKE なしでは読み出し不可。
    ↓ C2 復号
  メディア固有鍵 Kmu

Kmu + [ETK (暗号化タイトル鍵)] → SDカード 保護領域 SD_BIND/ に格納。AKE なしでは読み出し不可。
    ↓ C2 復号 (56bit 鍵)
  タイトル鍵 TK

TK + .sb1 暗号化ペイロード → C2-CBC 復号 → 平文 MPEG-4 AVC TS
```

**「コピー済みファイル単体で詰む」理由**: MKB・Media ID・ETK はいずれも CPRM 専用 SCSI コマンド（AKE）
経由でしかアクセスできない SDカード内の領域に存在する。通常の `cp`・`dd`・エクスプローラのコピーに
これらは含まれない。ETK のバイナリ（`file[0x0A0:0x0C0]`）は .sb1 内に存在するが、
これを復号するための Kmu が手元にない。

### 4-1. MEI006H サービスアーキテクチャ（解析で確定）

SD-MobileImpact が動作する環境において、**全ての CPRM 鍵演算は
`C:\Windows\SysWOW64\mei006h.exe`（Windows サービス）の内部で完結**する。
SD-MobileImpact ユーザープロセスのメモリには TitleKey も Kmu も一切現れない。

```
SD-MobileImpact.exe  (ユーザーモードプロセス)
        │ 名前付きパイプ \\.\pipe\mei006h
        ↓
MEI006H.exe  (SysWOW64 / Windows サービス)
        │
        ├── DeviceKey + MKB → Km
        ├── Km + MediaID → Kmu
        ├── Kmu + ETK → TitleKey (56bit)
        └── TitleKey + C2-CBC → 平文コンテンツ → パイプ経由で返す
```

- パイプ ACL: Authenticated Users に GRGWGX（読み書き実行）を許可するセキュリティ記述子を確認
- DLL ロードパターン: `%s\%s\%s\%016I64X%016I64X.DLL`
- 内部参照文字列: `\\.\pipe\mei006h`, `alterstar`, 複数 GUID
- **管理者権限なしではプロセスダンプ不可**（AccessDenied / error=5）

この設計により、SD-MobileImpact の全メモリダンプ（計 13 本以上）を対象に
TitleKey・Kmu の有無をスキャンしても **0 ヒット**となる（§6-2 参照）。

## 5. C2 (Cryptomeria) 暗号の実装解析

C2 暗号は CPRM が SD カードコンテンツ保護に採用する 64bit ブロック暗号である。
`SDVM2TSPacketDecParser.dll`（実行時展開型パック保護）のメモリダンプから F 関数・鍵スケジュールを逆解析した。

### 5-1. 確定した仕様

| 項目 | 値 |
|------|----|
| ブロック長 | 64bit (8B) |
| 鍵長 | 56bit (7B + 1B ゼロパディングで鍵スケジュール入力) |
| ラウンド数 | **16** |
| 構造 | XOR Feistel: `L, R ← R, F(R, K_r) ⊕ L` を 16 回、最終出力 `R‖L` |
| モード | C2-CBC (ユニット毎 IV リセット) |

### 5-2. ラウンド関数 F（VA 0x36014950 から逆解析）

```python
def F(W, K, ops):
    W = W ^ K                              # バイト単位 XOR（鍵挿入）
    W = S[W]                               # バイト単位 S-box 適用
    W[0] ^= W[1] ^ W[2] ^ W[3]            # 線形混合
    W[0] = S[W[0]]                         # W[0] 再 S-box
    W[1] = mix(ops[0], W[1], W[0])        # ラウンド固有演算（⊕ / + / −）
    W[2] = mix(ops[1], W[2], W[0])
    W[3] = mix(ops[2], W[3], W[0])
    return W

ROUND_OPS = [
    (XOR,SUB,ADD), (ADD,ADD,XOR), (ADD,XOR,ADD), (XOR,ADD,ADD),
    (SUB,XOR,ADD), (SUB,SUB,XOR), (SUB,ADD,XOR), (ADD,SUB,XOR),
    (SUB,ADD,ADD), (ADD,ADD,ADD), (ADD,SUB,ADD), (SUB,ADD,SUB),
    (ADD,ADD,SUB), (SUB,SUB,ADD), (ADD,SUB,SUB), (SUB,SUB,SUB),
]
```

### 5-3. 復号実装の注意点（バグの記録）

Feistel 暗号の復号では「ラウンド鍵の逆順」が必要なことは広く知られているが、
C2 ではラウンドごとに混合演算（ROUND_OPS）が変化するため、**ROUND_OPS も逆順**にする必要がある。

```python
# ❌ 誤り（当初の実装バグ）: 鍵は逆順だが ROUND_OPS は順方向
for r in range(16):
    Fr = F(R, K[15-r], ROUND_OPS[r])      # ← ROUND_OPS[r] が誤り → 復号失敗

# ✅ 正解: 鍵・ROUND_OPS の両方を逆順
for r in range(16):
    Fr = F(R, K[15-r], ROUND_OPS[15-r])   # ← 両方 15-r で正しい復号
```

`scripts/test_c2_roundtrip2.py` によるラウンドトリップ検証で確認済み。

## 6. スキャン結果（証拠）

### 6-1. スキャン方式

全 NumPy バッチ処理。dump 全体の 8B スライディングウィンドウを TitleKey 候補として試行。
オラクル条件: 6 連続暗号ブロック（CT[0]〜CT[5]）それぞれの復号結果の byte[0] または byte[4] が `0x47`（TS 同期バイト）。

### 6-2. 結果

| ダンプ | 採取タイミング | スキャン種別 | ヒット数 |
|--------|--------------|-------------|---------|
| dump_10 | コンテンツダブルクリック直後 | TitleKey 直接（R0+L0 両モード） | **0** |
| dump_11 | コンテンツ再生開始直後 | TitleKey 直接（R0+L0 両モード） | **0** |
| dump_11 | 〃 | Kmu→TitleKey 二段階（ETK×4候補、形式×3種） | **0** |
| dump_11 | 〃 | alt-oracle（逆方向 C2 暗号化オラクル） | **0** |
| dump_12 | コンテンツシーク後再生直後 | TitleKey 直接（R0+L0 両モード） | **0** |

全スキャンで 0 ヒット。これは §4-1 で述べた MEI006H アーキテクチャにより、
TitleKey・Kmu が SD-MobileImpact のユーザーモードメモリに存在しないことと整合する。

## 7. ワンセグ救出の実現方法

### 7-1. 環境

- ゲスト OS: Windows XP SP3 32bit
- VM: VMware Workstation Player（USB 2.0 パススルー）
- SD-MobileImpact: v1.0.61220.1000 (L032)（正規インストールメディア ISO より）
  - **L052 (Vista 対応版) に更新しない**: 追加モジュール配布が 2018/3 末に終了しており更新後に詰む

### 7-2. バイナリパッチ（86000012 エラー回避）

32GB SDHC / FAT32 を 2006 年版ソフトが「ファイルシステムが異なります（86000012）」と拒否する問題を
1バイト書換 × 2箇所で回避する:

| ファイルオフセット | 変更前 | 変更後 | 命令 | 効果 |
|---|---|---|---|---|
| `0x000429E9` | `75` | `EB` | `jne` → `jmp` | FS種別==0 判定ゲートをスキップ |
| `0x00078EBB` | `75` | `EB` | `jne` → `jmp` | 同上（別経路） |

```python
d = bytearray(open("SD-MobileImpact.exe", "rb").read())
for off in (0x429e9, 0x78ebb):
    assert d[off] == 0x75   # jne であることを確認
    d[off] = 0xeb           # jmp に書換
open("SD-MobileImpact_patched.exe", "wb").write(d)
```

※ version 1.0.61220.1000 専用のオフセット。他 version ではバイナリ内の即値 `12 00 00 86` を手がかりに再探索すること。

### 7-3. 黒画面の回避

VM 上でオーバーレイレンダリングが動作せず映像が黒画面になる（音声は正常）問題は、
ゲスト OS のハードウェアアクセラレータを「なし」に設定することで解消する:

> 画面のプロパティ → 設定 → 詳細設定 → トラブルシューティング → ハードウェアアクセラレータ = **なし**

### 7-4. 救出手順

1. BN-SDCMP3（元 microSD 装着）を USB 2.0 ハブ経由でホストに接続
2. VMware の「このコンピュータから切断」でゲスト XP にパススルー（ゲスト側に F: として出現）
3. パッチ版 EXE を管理者権限で起動 → 動画タブで F: を選択 → 番組一覧が表示される
4. 番組を再生 → 映像・音声が出ることを確認
5. ホスト側 OBS Studio でウィンドウ（または全画面）とデスクトップ音声を録画
6. 全 18 番組を 1 本ずつ再生して録画（総計 約 40 分）

⚠️ **絶対に行ってはならない操作**: SDフォーマット / SDへの書き戻し / 録画データの削除  
（CPRM は Media ID 紐付きのため、誤操作でオリジナルカードのデータが失われると復元不可能）

## 10. 将来の ISDB 規格への提言

詳細: [`doc/preservation_policy.md`](doc/preservation_policy.md)

**2026 年時点での現状**: ワンセグ/フルセグ録画を正規手段で救出する方法は事実上存在しない。
唯一の現実的経路は「Windows XP SP3 VM + 2006 年版ソフト + バイナリパッチ」であり、
これは今後さらに困難になる（VM ソフトの XP 対応終了・Internet Archive のファイル削除等）。

同じ設計上の失敗を ISDB-S3（BS 4K/8K・2018 年放送開始済み）および
地上波 4K（ISDB-T3 計画中）の録画保護設計で繰り返すべきでない。

主な提言（5 項目）:
1. 機器廃棄・サービス終了時の正規移行手段の義務化
2. DRM ソフトウェアの最終出荷後 10 年間維持義務 / 不可能な場合の鍵エスクロー
3. 「自己録画コンテンツの保存目的」を著作権法上で明示的に免除
4. コンテンツをオープンコンテナ（MPEG-TS）に格納し DRM を鍵管理層に分離
5. 正規再生手段が消滅した録画コンテンツの公的アーカイブ支援制度の創設
