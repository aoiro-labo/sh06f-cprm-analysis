# 暗号方式・ファイル構造 技術解析

最終更新: 2026-06-19 ／ 対象: SH-06F の `.sb1`(ワンセグ) と `FSEG`本体(フルセグ)

> 目的：復号鍵が手元に無い前提で、**暗号方式とファイル構造をリバースエンジニアリングで特定**する。
> 実機・カードが無くても進められる「中身の解析」の記録。

---

## 1. ワンセグ `MOV001.sb1` の構造（実測で確定）

### 1-1. 全体レイアウト
- **ファイル長は必ず 2048 (0x800) の倍数**（全18本で確認）。記録は2KBセクタ単位。
- さらに **6144 バイト (0x1800 = 3×2KB) ごとの「ユニット」** で周期構造を持つ。
- 各ユニットの先頭 **160バイト (0xA0) は平文ヘッダ**、残り **5984バイトが暗号化ペイロード**。

```
1ユニット = 6144 B
├ 0x000–0x09F (160B)  平文ヘッダ（下記 1-2）
└ 0x0A0–0x17FF(5984B) 暗号化ペイロード   ← 5984 = 748 × 8
```

### 1-2. 平文ヘッダ（0xA0）の中身
全18録画で**先頭0xA0が完全に同一**(md5 `b7e03c76a361…`)。＝個別鍵情報は無い固定テンプレート。

```
0x00: 00 00 00 00                            （4バイトのパディング/タイムスタンプ枠）
0x04: 47 00 1F 1X 00 80 30 B4 C0 …           （TS類似マーカー。X=連続性カウンタ 0,1,2…）
0x60: 08 00 1F 00                            （副マーカー）
0x9F: 39                                     （固定フラグ）
0xA0: ここから暗号化
```
- `47` = MPEG-2 TS の同期バイト風。`00 1F` = PID 0x001F 相当。`1X` の下位4bitが
  ユニットごとに 0→1→2… と増える＝**TS連続性カウンタ**。
- ただし長さが160Bで188Bのフルパケットではない＝**SHARP独自のユニット区切りヘッダ**。

### 1-3. 暗号モードの判定（実測）
| 観測 | 結果 | 意味 |
|------|------|------|
| 8Bブロックの重複(全域) | ほぼ皆無(最大74回=ユニット数と一致) | ECBではない |
| 各ユニット先頭の暗号ブロック(0xA0,0xA8) | 74ユニット全てで**完全一致** | **ユニット毎にIVリセット＋先頭平文が一定** |
| 暗号領域長 5984 | **8で割り切れる**(748) | **ブロック長8バイト=64bit暗号** |
| 18録画の先頭暗号ブロック | **全て異なる(18/18)** | **タイトル鍵は録画ごとに別** |

→ **結論：64bitブロック暗号の CBC モード、6144Bユニット毎にIV(変換データ)をリセット。**
　各ユニットの平文先頭16B以上が固定（PES/区切りヘッダ）なので、CBCの性質で
　先頭2ブロックの暗号文がユニット間で一致する。これは **CPRM の C2(Cryptomeria) 暗号
　＝C2-CBC（変換CBC）** の挙動と一致する。

### 1-4. ここから言える攻撃面（重要）
- 各ユニット先頭は **既知平文(固定ヘッダ)×固定IV** の状態 → **既知平文ペアが採れる**。
- C2 の鍵は **56bit**。よって理論上の最短攻撃は「**56bit鍵の総当たり＋既知平文照合**」。
  - 必要物: ①C2の秘密S-box(公開実装 "libc2"/Cryptomeria に復元済みのものがある)
            ②正しい既知平文(固定ヘッダの中身の特定)
  - コスト: 2^56 ≒ DES並み。家庭用PCでは非現実的(専用FPGA/GPU群で数日〜)。録画ごとに別途必要。
- カードの保護領域・デバイス鍵は一切不要なのがこの経路の利点だが、規模が大きい。

## 2. フルセグ `0000100010` の構造（実測）

| 項目 | 値 |
|------|----|
| サイズ例(10001) | 883,028,160 B。**192で割り切れる**(4,599,105)が2048では割り切れない |
| パケット構造 | **192バイト = TTS/M2TS(4Bタイムスタンプ+188B TS)** |
| 平文ヘッダ | **無し**（2KB境界に構造ヘッダが出ない） |
| 0x47同期 | 検出されず＝**TSヘッダごと全域暗号化** |
| 8Bブロック重複(先頭8MB) | **ゼロ**（104万ブロック全てユニーク） |
| 192Bパケット先頭ブロック | ほぼ全ユニーク（リセット痕跡なし） |
| 録画ごとの先頭ブロック | 全て異なる＝**録画ごとに別鍵** |

→ **結論：フルセグは連続チェイン暗号(CBC/CTRを止めずに流す)。** ワンセグと違い
　ユニット毎リセットや固定ヘッダの露出が無いため、**既知平文クリブが得られず**、
　ワンセグより更に攻撃が困難。DBの `secureInfo`/`emmNum`(各64B)が鍵・権利情報だが
　SH-06F本体内の秘密鍵が無いと復号できない。

## 3. 鍵階層と「手元のコピーで詰む」理由（再掲・技術版）

```
[デバイス鍵(秘密)] + [MKB(カードのシステム領域)]      → メディア鍵 Km
Km + [Media ID(カード読取専用領域)]                  → メディア固有鍵 Kmu
Kmu で [暗号化タイトル鍵(保護領域 SD_BIND)] を復号  → タイトル鍵(56bit C2鍵)
タイトル鍵 + C2-CBC でコンテンツ復号
```
- 本ケースで暗号文(`.sb1`/FSEG)と一部の暗号化メタ(576B/64B BLOB)は手元にある。
- **欠けているのは Media ID・MKB・暗号化タイトル鍵**＝全てカードの保護/読取専用領域。
  通常コピーには物理的に入らない（CPRM認証コマンド経由でしか読めない）。
- 「576Bファイル」は録画1・2で同一、3・4で各々別。→ カード状態依存の鍵ブロックと推定。
  単体では Kmu が無いと復号不能。

## 4. 現実的な技術ルート（カード/実機が使える場合）

復号の本筋は「鍵を持っている主体（カード保護領域 or 実機）から鍵を取る」こと。

1. **StationTVアプリのネイティブ実装を解析**
   - 今あるAPKフォルダは**リソースのみ抽出版**（assets/res/arscのみ、`classes.dex`も`lib/*.so`も無し）。
   - 必要物：**実機 or 配布元の本物の `.apk`**。展開して `lib/armeabi*/*.so`(NDK実装) を見れば、
     復号がアプリ内か `/system` のSHARP製DRMライブラリ委譲かが判る。
   - 期待値：アルゴリズム確認には有用。ただし鍵自体は実機のセキュア領域にある可能性大。

2. **SH-06F のシステムライブラリ/セキュア領域**
   - SHARPのCPRM/CAS実装(例 `libtvcas`系)と**デバイス鍵**は本体側。root化して
     鍵ラダーを再現できれば、自前で Kmu→タイトル鍵→C2復号 が原理的に可能。
   - ハードルは高い（鍵がTrustZone/ハード内なら取り出せない）。

3. **56bit総当たり（カード不要だが大規模）**
   - 1章の通りワンセグは既知平文クリブが採れる。固定ヘッダ平文を特定できれば
     C2鍵(56bit)を総当たり可能。専用ハードが要る規模。フルセグはクリブ無しで不可。

## 5. C2(Cryptomeria)実装の確定事項（2026-06-23 追記）

### 5-1. ラウンド数・構造

- **ラウンド数: 16**（`ROUND_OPS` テーブルが 16 エントリ、C2解読_ダンプRE.md §2 でも"計16ラウンド鍵"と確認）
  ※ 旧記述「10ラウンド」は誤り。正しくは **16ラウンド**。
- **構造**: XOR Feistel。`L,R ← R, F(R, K_r) ⊕ L` を16回。最終出力は `R||L`（スワップ済み）。
- **ラウンド関数 F**: バイト単位 S-box 適用 → 線形混合 → ラウンド固有混合演算 (⊕/+/-)。
  混合演算は `ROUND_OPS[r]` で決まり、ラウンドごとに異なる。

### 5-2. 復号の正しい実装（重要）

C2 復号は鍵の逆順だけでなく **`ROUND_OPS` も逆順** にする必要がある。

```python
# ❌ 誤り (初期実装のバグ): 鍵は逆順だが ROUND_OPS は順方向のまま
for r in range(16):
    Fr = F(R, K[15-r], ROUND_OPS[r])   # ROUND_OPS が forward → 復号失敗

# ✅ 正解: 鍵も ROUND_OPS も両方逆順
for r in range(16):
    Fr = F(R, K[15-r], ROUND_OPS[15-r])  # 両方 15-r → 正しい復号
```

`scripts/test_c2_roundtrip2.py` で検証済み（全ラウンドトリップ PASS）。

### 5-3. MEI006H アーキテクチャ（2026-06-23 確定）

**全ての CPRM 鍵演算は Windows サービス `C:\Windows\SysWOW64\mei006h.exe` の内部で完結**。
SD-MobileImpact のユーザーモードプロセスには TitleKey も Kmu も現れない。

```
SD-MobileImpact.exe (ユーザーモード)
    ↓ 名前付きパイプ \\.\pipe\mei006h
MEI006H.exe (Windows サービス / SysWOW64)
    → DeviceKey + MKB → Km → Kmu → TitleKey → C2-CBC 復号
    → 平文コンテンツをパイプ経由で返す
```

証拠:
- 全 4 種スキャン（TitleKey 直接・Kmu 二段階・逆方向・複数ダンプ）で **0 ヒット**
- `mei006h.exe` 内に `\\.\pipe\mei006h`・`alterstar`・複数 GUID を確認
- パイプ ACL: Authenticated Users に GRGWGX（読み書き実行）を許可

### 5-4. スキャン結果サマリ（証拠）

| スキャン種別 | 対象ダンプ | ヒット数 |
|-------------|-----------|---------|
| TitleKey 直接 (R0+L0) | dump_10, 11, 12 | **0** |
| Kmu → TitleKey 二段階 (ETK×4候補, 形式×3) | dump_11 | **0** |
| alt-oracle (逆方向オラクル) | dump_11 | **0** |

→ SD-MobileImpact ユーザーモードメモリに鍵材料が存在しないことを多角的に確認。
