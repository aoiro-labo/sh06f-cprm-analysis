# 暗号方式・ファイル構造 技術解析

最終更新: 2026-06-23 ／ 対象: SH-06F の `.sb1`(ワンセグ) と `FSEG`本体(フルセグ)

> 目的：SH-06F 実機・SD カードを使いながら、暗号方式・鍵チェーン・ライブラリ構造を
> リバースエンジニアリングで特定し、PC での復号実現の可否と経路を整理する。

---

## 1. ワンセグ `MOV001.sb1` の構造（実測で確定）

### 1-1. 全体レイアウト
- **ファイル長は必ず 2048 (0x800) の倍数**（全18本で確認）。記録は2KBセクタ単位。
- さらに **6144 バイト (0x1800 = 3×2KB) ごとの「ユニット」** で周期構造を持つ。
- 各ユニットの先頭 **160バイト (0xA0) は平文ヘッダ**、残り **5984バイトが暗号化ペイロード**。

```
1ユニット = 6144 B
├ 0x000–0x09F (160B)  平文ヘッダ（下記 1-2）
└ 0x0A0–0x17FF(5984B) 暗号化ペイロード   ← 5984 = 748 × 8
```

### 1-2. 平文ヘッダ（0xA0）の中身
全18録画で**先頭0xA0が完全に同一**(md5 `b7e03c76a361…`)。＝個別鍵情報は無い固定テンプレート。

```
0x00: 00 00 00 00                            （4バイトのパディング/タイムスタンプ枠）
0x04: 47 00 1F 1X 00 80 30 B4 C0 …           （TS類似マーカー。X=連続性カウンタ 0,1,2…）
0x60: 08 00 1F 00                            （副マーカー）
0x9F: 39                                     （固定フラグ）
0xA0: ここから暗号化
```
- `47` = MPEG-2 TS の同期バイト風。`00 1F` = PID 0x001F 相当。`1X` の下位4bitが
  ユニットごとに 0→1→2… と増える＝**TS連続性カウンタ**。
- ただし長さが160Bで188Bのフルパケットではない＝**SHARP独自のユニット区切りヘッダ**。

### 1-3. 暗号モードの判定（実測）
| 観測 | 結果 | 意味 |
|------|------|------|
| 8Bブロックの重複(全域) | ほぼ皆無(最大74回=ユニット数と一致) | ECBではない |
| 各ユニット先頭の暗号ブロック(0xA0,0xA8) | 74ユニット全てで**完全一致** | **ユニット毎にIVリセット＋先頭平文が一定** |
| 暗号領域長 5984 | **8で割り切れる**(748) | **ブロック長8バイト=64bit暗号** |
| 18録画の先頭暗号ブロック | **全て異なる(18/18)** | **タイトル鍵は録画ごとに別** |

→ **結論：64bitブロック暗号の CBC モード、6144Bユニット毎にIV(変換データ)をリセット。**
　各ユニットの平文先頭16B以上が固定（PES/区切りヘッダ）なので、CBCの性質で
　先頭2ブロックの暗号文がユニット間で一致する。これは **CPRM の C2(Cryptomeria) 暗号
　＝C2-CBC（変換CBC）** の挙動と一致する。

### 1-4. ここから言える攻撃面（重要）
- 各ユニット先頭は **既知平文(固定ヘッダ)×固定IV** の状態 → **既知平文ペアが採れる**。
- C2 の鍵は **56bit**。よって理論上の最短攻撃は「**56bit鍵の総当たり＋既知平文照合**」。
  - 必要物: ①C2の秘密S-box(公開実装 "libc2"/Cryptomeria に復元済みのものがある)
            ②正しい既知平文(固定ヘッダの中身の特定)
  - コスト: 2^56 ≒ DES並み。家庭用PCでは非現実的(専用FPGA/GPU群で数日〜)。録画ごとに別途必要。
- カードの保護領域・デバイス鍵は一切不要なのがこの経路の利点だが、規模が大きい。

## 2. フルセグ `0000100010` の構造（実測）

### 2-1. AV ストリーム本体

| 項目 | 値 |
|------|----|
| サイズ例(10001) | 883,028,160 B。**192で割り切れる**(4,599,105)が2048では割り切れない |
| パケット構造 | **192バイト = TTS/M2TS(4Bタイムスタンプ+188B TS)** |
| 平文ヘッダ | **無し**（ファイル先頭から全域暗号化） |
| 0x47同期 | 先頭4KB・中間256B・末尾256Bいずれも検出されず＝**TSヘッダごと全域暗号化** |
| 8Bブロック重複(先頭8MB) | **ゼロ**（104万ブロック全てユニーク） |
| 192Bパケット先頭ブロック | ほぼ全ユニーク（ユニット毎 IV リセット痕跡なし） |
| 録画ごとの先頭8B | 全4録画で異なる＝**録画ごとに鍵またはIVが異なる** |
| エントロピー(256Bサンプル) | 先頭7.22 / 中間7.09〜7.23 / 末尾7.14〜7.27（全域高エントロピー） |

→ **結論：フルセグは連続チェイン暗号。** ワンセグと違いユニット毎リセットや
　固定ヘッダの露出が無いため、**既知平文クリブが得られず**、ワンセグより攻撃が困難。

### 2-2. 576B 鍵ブロック（`0001100010`〜`0001100040`）詳細構造（2026-06-23 確定）

576B ファイル全4録画のバイト単位比較から以下の構造が確定した。

```
0001100010 = [固定部 (272B)] + [可変部 (304B)]

固定部: offset 0x000〜0x10F (272B)
  → 全4録画で完全一致（MD5 不変）
  → 先頭8B: c5 52 87 a2 47 d1 c1 46
  → 可能性: デバイスID / カード識別 / 認証ブロック（詳細フォーマット未解明）
  → ゼロ連続なし・エントロピー高い → 全域何らかの暗号化または固定係数

可変部: offset 0x110〜0x23F (304B = 19×16B AESブロック相当)
  → 録画ごとに異なる
  → 10001 と 10002 は可変部も含め全576Bが完全一致（MD5同一）
  → 10003・10004 はそれぞれ異なる
```

| 録画 | MD5 | 可変部の値 |
|------|-----|-----------|
| 10001 | d9402b8d... | 同一グループ A |
| 10002 | d9402b8d... | 同一グループ A（10001と完全一致）|
| 10003 | 0e3c1db3... | グループ B |
| 10004 | 31749ec3... | グループ C |

**10001 と 10002 のタイトル鍵が同一かどうか**:
AV ストリームの先頭8Bは 10001 と 10002 で異なる（`7e6d73e7...` vs `2099 01b1...`）。
鍵ブロックが同一でも先頭暗号文が異なる → CBC の IV が録画ごとに変わっているか、
または暗号化の別レイヤーで IV が挿入されている。

### 2-3. SQLite BLOB（secureInfo / emmNum / copyCount）構造

`SHFSEG0001.DB` の `fseg` テーブルスキーマ（PRAGMA table_info で確認）:

| 列番号 | 名前 | 型 | 備考 |
|--------|------|-----|------|
| 0 | contentsId | INTEGER | 録画ID (10001〜) |
| 1 | title | TEXT | Shift-JIS |
| 4 | recStartDate | TEXT | `YYYY/MM/DD HH:MM:SS` |
| 5 | recEndDate | TEXT | 同上 |
| 6 | playTime | INTEGER | 再生時間 (ms) |
| 7 | dataSize | INTEGER | AV ファイルサイズ (B) |
| 8 | copyCount | BLOB | 64B |
| 12 | secureInfo | BLOB | 64B ← デバイス鍵で AES 暗号化されたコンテンツ鍵 |
| 13 | emmNum | BLOB | 64B |

4録画の実データ（抜粋）:

| 録画 | タイトル | 局 | 開始〜終了 | 時間 | サイズ |
|------|---------|-----|-----------|------|--------|
| 10001 | まどか☆マギカ TV Edition | CBC テレビ | 12/07 17:20〜17:30 | 559s | 842MB |
| 10002 | バナナサンド☆冬ソング | CBC テレビ | 12/09 20:07〜20:17 | 602s | 1195MB |
| 10003 | うたコン アワード2025 | NHK総合 | 12/09 20:18〜20:28 | 611s | 1176MB |
| 10004 | 今夜はナゾトレ | 東海テレビ | 12/09 20:30〜20:41 | 619s | 1188MB |

各 64B BLOB の 16B 単位エントロピー: 3.875〜4.000（**AES-128 出力の典型値**）。  
4録画で全て異なる → 録画固有の情報が含まれる。  
`KB[0x110:0x150] XOR secureInfo` エントロピー ≈ 5.75 → 直接的な XOR 関係なし。

- `secureInfo (64B)`: **デバイス鍵で AES 暗号化されたコンテンツ鍵（AES Key + IV）**
- `emmNum (64B)`: EMM 関連（CAS 系）
- `copyCount (64B)`: コピー制御情報（暗号化）
- `576B 鍵ブロック可変部 (304B)`: デバイス/カードバインディング情報

## 3. 鍵チェーン整理

### 3-1. ワンセグ（CPRM）

```
[デバイス鍵] + [MKB (SD カードシステム領域)]   → メディア鍵 Km
Km + [Media ID (SD カード読取専用領域)]         → メディア固有鍵 Kmu
Kmu + [ETK (SD カード保護領域)]                → タイトル鍵 (56bit C2 鍵)
タイトル鍵 + C2-CBC → コンテンツ復号
```

手元にあるもの: `.sb1`（暗号文）  
**欠けているもの**: MKB・Media ID・ETK（SD カードの保護/読取専用領域）。  
SD カードを実機に挿して CPRM 認証を通さないと取得不可。

SD-MobileImpact 経由では Windows サービス `mei006h.exe` 内で全鍵演算が完結し、  
ユーザーモードプロセスに TitleKey・Kmu は一切現れない（メモリダンプで確認済み）。

### 3-2. フルセグ（AES + デバイス鍵）

```
[デバイス鍵 (/system/etc/mccd/sbdb)]
    ↓ AES-128 復号
secureInfo (64B, SHFSEG0001.DB)  →  AES Key + IV
    ↓ AES-128-CBC 復号
SD カード AV ファイル (0000100010 等)  →  平文 M2TS
```

手元にあるもの: AV ファイル（暗号文）、secureInfo（AES で暗号化された鍵）  
**欠けているもの**: デバイス鍵（`/system/etc/mccd/sbdb`、root 専用）  

576B 鍵ブロック (`0001100010` 等) は librmp.so が管理する RMP Key Block と推定。  
デバイスの ME 鍵で保護。device key が取れれば secureInfo → AES Key → 復号の経路は明確。

| ルート | 対象 | 難度 | 状況 |

## 5. C2(Cryptomeria)実装の確定事項（2026-06-23 追記）

### 5-1. ラウンド数・構造

- **ラウンド数: 16**（`ROUND_OPS` テーブルが 16 エントリ、C2解読_ダンプRE.md §2 でも"計16ラウンド鍵"と確認）
  ※ 旧記述「10ラウンド」は誤り。正しくは **16ラウンド**。
- **構造**: XOR Feistel。`L,R ← R, F(R, K_r) ⊕ L` を16回。最終出力は `R||L`（スワップ済み）。
- **ラウンド関数 F**: バイト単位 S-box 適用 → 線形混合 → ラウンド固有混合演算 (⊕/+/-)。
  混合演算は `ROUND_OPS[r]` で決まり、ラウンドごとに異なる。

### 5-2. 復号の正しい実装（重要）

C2 復号は鍵の逆順だけでなく **`ROUND_OPS` も逆順** にする必要がある。

```python
# ❌ 誤り (初期実装のバグ): 鍵は逆順だが ROUND_OPS は順方向のまま
for r in range(16):
    Fr = F(R, K[15-r], ROUND_OPS[r])   # ROUND_OPS が forward → 復号失敗

# ✅ 正解: 鍵も ROUND_OPS も両方逆順
for r in range(16):
    Fr = F(R, K[15-r], ROUND_OPS[15-r])  # 両方 15-r → 正しい復号
```

`scripts/test_c2_roundtrip2.py` で検証済み（全ラウンドトリップ PASS）。

### 5-3. Windows 側（SD-MobileImpact / ワンセグ）の鍵管理

全 CPRM 鍵演算は Windows サービス `C:\Windows\SysWOW64\mei006h.exe` 内で完結。  
SD-MobileImpact のユーザーモードプロセスには TitleKey も Kmu も現れない。  
（メモリダンプを全 4 種スキャン、ヒット 0 で確認。）

```
SD-MobileImpact.exe → 名前付きパイプ \\.\pipe\mei006h → MEI006H.exe
  → DeviceKey + MKB → Km → Kmu → TitleKey → C2-CBC 復号 → 平文を返す
```

## 6. フルセグ復号チェーン詳細（Android ライブラリ逆解析・2026-06-23）

SH-06F から ADB 経由でプル・解析した 26 本のシステムライブラリにより、
フルセグの暗号チェーンが以下のように確定した。

### 6-1. 暗号化ライブラリの呼び出し構造

```
StationTV (jp.pixela.stationtv.*)
    ↓ JNI
libshfsegsavejni.so
    ↓ C++ API
libshfsegsave.so  (shFSegClient クラス)
    ├── libshfsegsave_Common.so   (パス管理・DB操作)
    ├── libshfsegsave_CustomSH.so (SHARP固有実装・SDカード検出)
    ├── libshfsegsave_SDMiddle.so (SD状態管理)
    └── libshfsegsave_Crypt.so    ← 暗号化/復号の実体
            ↓ 内部呼び出し
        libsceseccnt.so  (SCEMID API 実装)
            SCEMID_SecCntPreDecode
            SCEMID_SecCntDecode
            SCEMID_SecCntPostDecode
```

再生時は別経路が加わる:
```
libMmbFcLiceMw.so  (フルセグライセンスMW)
    ├── MmbFcLiceMwUseLicence::requestPlayStart()
    │       getSecureItem()          ← DB から secureInfo(64B) 取得
    │       MmbPoAesMp_Decrypt()     ← デバイス鍵で secureInfo 復号
    │       → AES Key + IV を取得
    │       MmbFcLiceMwFileCipher::setAesParamater(mode, key, IV, ...)
    │           MmbPoAesMp_Decrypt() ← コンテンツをブロック単位で復号
    └── MmbCaCasDrmMwClient (socket通信)
            接続先: /dev/socket/MmbCaCasDrmMw  ← world-writable!
            authenticate() → ECM/EMM 処理
            getRecSecData() → 録画セキュリティデータ取得
```

### 6-2. デバイス固有鍵の格納場所

| パス | 内容 | アクセス権 |
|------|------|----------|
| `/system/etc/mccd/sbdb` | AES 鍵（`MmbPoSstMp_GetAESKey`が参照） | `drwx------ root root` → root のみ |
| `/system/etc/mccd/sbro` | バックアップ鍵？ | 同上 |
| `/persist/data` | CPRM プロビジョニングデータ | 同上 |
| `/data/DxDrm/DxDeviceKey` | Discretix DRM デバイス鍵 | 存在しない（未プロビジョニング） |

`getLocalEncKey()` (`MmbStRecMwWrite`) の呼び出しシーケンス:
1. `MmbPoSstMp_GetNvItem()` → Qualcomm NV ストレージから NV アイテム取得
2. `MmbPoSstMp_GetAESKey()` → `/system/etc/mccd/sbdb` からAES鍵読み込み
3. `MmbPoAesMp_Encrypt()` → TS ストリームをAES暗号化して書き込み

### 6-3. RMP (Rights Management & Protection) ライブラリ

```
libshfullseg_keyprov.so
    shfullseg_write_provisioning()  ← RMP プロビジョニングデータ書き込み
    shfullseg_verify_provisioning() ← プロビジョニング検証
        ↓
    librmp.so  (ARIB STD-B61 RMP 実装)
```

576B 鍵ブロックファイル (`0001100010` 等) は `librmp.so` が管理する
**RMP Key Block (RMPKB)** と推定。
デバイスの ME 鍵（Mobile Equipment Key）で保護されており、
ME 鍵なしにはコンテンツ鍵を取り出せない。

### 6-4. DRM ソケット `/dev/socket/MmbCaCasDrmMw`（調査中）

パーミッション `srw-rw-rw-`（world read/write）。サーバー: `MmbFcLiceMwServer`（root, PID 290）。

サーバーの公開 API（`libMmbFcLiceMw.so` 解析より）:
- `getRecSecData(unsigned char*)` → 録画セキュリティデータ（コンテンツ鍵？）
- `processEcm()` / `updateEmm()` → CAS 鍵管理
- `requestPlayStart/End()` → 再生制御

**認証プロトコル**（`MmbCaCasDrmMwCasSocket::authenticate()`, 1068B を逆解析）:
- `MmbPoRandMp_RandBytes` で 32B のノンスを生成
- `sendData<MmbAuthenticateReqParam>` でサーバーに送信
- `unlink` + `pthread_create` のパターン → クライアントが折り返し用ソケットを作成する可能性
- バイナリ中に定数 `11 22 33 44 55 66 77 88 99 AA BB CC DD EE FF`（15B）が `authenticate()` 直前に存在

**認証プロトコル全体像（2026-06-24 確定）**:

DRM ソケットサーバー実体は `/system/bin/MmbCaCasDrmMw`（PID 286、root）。
Abstract socket `@MmbCaCasDrmMw` をリッスン（ファイルシステム側 `/dev/socket/MmbCaCasDrmMw` ではない）。

```
接続 → サーバーが待機
[1] Client: FT=0x00, DataLen=32(LE), 32B nonce
[2] Server: FT=0x00, DataLen=32(LE), 32B challenge
[3] Client: FT=0x00, DataLen=32(LE), 32B response ← ここで検証
Server: 不正なら line 491 "Authenticate Data - error"
```

**response の計算には device key が必要**:
`/durable/etc/mccd/DB001.bin`（root 専用、`drwx--x--x` 配下）から読み出した 16B 鍵を使用。
AES/hash の具体的な計算式は逆解析中だが、device key なしでは応答が計算不可能。

**APK インストールによるアクセステスト（2026-06-24）**:
`com.test.drmtest`（UID 10195）から `@MmbCaCasDrmMw` に接続 → `ECONNREFUSED`。
→ SELinux ポリシーが `untrusted_app:s0` から `device:s0` ソケットへの connect を拒否。
ADB daemon (`adbd:s0`, UID 2000) は接続できる（0.7ms close は FILESYSTEM 側の非リッスン問題）。

### 6-5. デバイス鍵の取得経路（2026-06-23 確定）

**`MmbPoSstMp_GetAESKey`** は `libMmbStRecMw.so` に内部実装されており、  
`/system/etc/mccd/sbdb`（root 専用）を直接 `fopen` して AES 鍵を読む。

StationTV（UID 10068）がこの関数を呼べる経路は現時点で不明。
可能性: `MmbStCtlMwService`（root, PID 427）が提供するローカルソケット経由だが、
そのソケット（`/data/mmb/common/streccm/MMBSTRECCMMW_SOCKET`）は  
`drwxrwx--- root mmb` → ADB shell からはアクセス不可。

**`fclicemwserver` Binder サービス**（shell から呼び出し可能）:
- `getSecureItem(crid)` → DRM ライセンスメタ（startTime, usableSec 等）のみ返す
- AES コンテンツ鍵は含まない（メモリ内リストは再起動でクリア）
- AES 鍵の復号は CLIENT 側で `MmbPoAesMp_Decrypt(secureInfo, device_key)` として実行
- `libMmbPoAesMp.so` = OpenSSL の `AES_cbc_encrypt/decrypt` 純粋ラッパー（サーバー委譲なし）

### 6-6. 取得済みライブラリ（主要なもの）

`PRIVATE/android_libs/` に ADB pull 済:
`libshfsegsave_Crypt.so`, `libshfsegsave.so`, `libshfsegsave_Common.so`,

## 8. Dirty COW / GetHash インジェクション実験（2026-06-26〜27）

### 8-1. CVE-2016-5195 (Dirty COW) の有効性確認

SH-06F カーネル 01.01.07 (kernel 3.4.0) において **CVE-2016-5195 が未パッチであること**を
カーネルソース (`mm/memory.c`) 解析で確認。

- towelroot (CVE-2014-3153): PATCHED
- CVE-2015-3636: PATCHED
- CVE-2016-5195: **NOT PATCHED**

`dirtycow2` でテスト: `/system/bin/run-as` のページキャッシュへの書き込みに成功。  
ただし `run-as` に SUID ビットなし → 単純実行ではルート権限取得不可。

### 8-2. Dirty COW によるライブラリ注入（libMmbPoHashMp.so）

`MmbPoHashMp_GetHash`（VA 0x828, size 0x64=100B, libMmbPoHashMp.so）を注入ターゲットに選定。  
理由：MmbCaCasDrmMw 起動後の初回認証まで **I-cache がコールド** → 注入コードが実行される。

```
注入手順:
1. dirtycow4 (dirtycow_lib.c 流用) で libMmbPoHashMp.so のオフセット 0x828 を書き換え
2. DRM ソケットに接続してフル認証フロー (nonce → challenge → fake response) を送信
3. GetHash が呼ばれたタイミングでシェルコードが実行
4. 結果を fds 3..50 の write() で TCP ソケット経由で返送
```

注入サイズの制約: libMmbPoHashMp.so の GetHash は 100B。次の関数 GetHashFromFile (226B)、
GetHmac (138B)、Mmb_CheckLog (108B) の順に配置。**Mmb_CheckLog (offset 0x1D0) 以上に
はみ出すとクラッシュ**（認証中にログ出力が発生するため）。実用的な上限は **約 200B**。

### 8-3. GetHash 引数の観測結果

**GetHash 署名（推定）**: `MmbPoHashMp_GetHash(int type, void *input, int len, void *output)`

認証フロー中の観測値（複数セッション分）:

| フィールド | 値 |
|-----------|-----|
| type | 1 (おそらく SHA1 または独自ハッシュ) |
| in_len | 48 バイト |
| input B0-15 | セッションごとに異なる（中間ハッシュ値、認証毎に変わる） |
| input B16-31 | **2パターンで交互出現**（定数、下記参照） |
| input B32-47 | ゼロ埋め |

**B16-31 の定数パターン（全セッション共通）**:

```
Pattern A: 11 22 33 44 55 66 77 88 99 00 11 22 33 44 55 66
Pattern B: 11 22 33 44 55 66 77 88 99 aa bb cc dd ee ff 00
```

各認証セッション内で GetHash はペアで呼ばれ、同じ B0-15 に対して
Pattern A / Pattern B が交互に使われる。B0-15 はセッションごとに変化する。

**解釈**: B16-31 は鍵ではなくプロトコル固定値（ティア情報またはカウンタ）と推定。
デバイス鍵は GetHash の引数として現れない（内部的に HMAC キーとして使用）。

### 8-4. GetHash シェルコード経由でのファイルアクセス結果

GetHash シェルコードから DRM プロセス (root, 各種 SELinux コンテキスト) でファイルアクセスを試みた結果:

| パス | open() 結果 | 内容 |
|------|------------|------|
| `/durable/etc/mccd/TempDB001.bin` | ENOENT | SELinux が存在を隠蔽 |
| `/durable/etc/mccd/DB001.bin` | ENOENT | 同上 |
| `/durable` (ディレクトリ) | EACCES | 存在するがアクセス不可 |
| `/system/etc/mccd/sbdb.rb.db` | ENOENT | ファイルなし |
| `/system/etc/mccd/sbdb` | **fd=11、成功** | 36B (下記参照) |
| `/system/etc/mccd/sbro` | **fd=11、成功** | 64B (下記参照) |

**注意**: ENOENT は SELinux の "dontaudit + 存在隠蔽" 動作と思われる

### 8-6. sbdb を使ったフルセグ復号試行（失敗）

`sbdb[0:16]` および各種組み合わせで `secureInfo (64B)` の AES-128 復号を試みた:

| 試行 | 操作 | M2TS sync hits |
|------|------|---------------|
| sbdb[0:16] ECB | AES-128-ECB(key=sbdb_b0, data=secure[0:16]) | 0/100 |
| sbdb[16:32] ECB | AES-128-ECB(key=sbdb_b1, ...) | 0/100 |
| sbdb_b0 XOR sbdb_b1 | XOR of two blocks | 0/100 |
| AES(sbdb_b0, sbdb_b1) | sbdb_b1 を sbdb_b0 で復号 | 1/100 (ノイズ) |
| sbdb_b0 (CBC, iv=zeros) | AES-128-CBC | 0/100 |
| sbdb(AES-256, b0+b1) | AES-256 | ≤1/100 |
| sbro 4ブロック各種 | 16B 単体・XOR 組み合わせ | 0/100 |
| バイト逆順・エンディアン変換 | 各種変換 | ≤2/100 (ノイズ) |

**結論**: sbdb の内容はそのままでは AES デバイス鍵として機能しない。  
考えられる原因:
1. sbdb は参照元 (`sbdb.rb.db`) のスタブで、本体は存在しない
2. deivce key は sbdb ではなく TrustZone / ハードウェアキーラダー経由で提供
3. 鍵導出にさらなる中間処理（CPRM KDF など）が必要

### 8-7. AES_Decrypt シェルコードの現状

`libMmbPoAesMp.so` のオフセット 0x7F4 (`MmbPoAesMp_Decrypt`, Thumb モード確認済み) に
引数キャプチャシェルコードを注入済み（2026-06-27 時点）。

シェルコードは:
- r0 (type)、r1 (ptr1 の先頭 16B)、r2 (len)、r3 (ptr2 の先頭 16B) を socket fd 経由で返送

**次のステップ**: 端末でフルセグ録画を再生開始 → AES_Decrypt が実際のコンテンツ鍵で呼ばれるタイミングで引数を取得。

### 8-8. 現在の注入済み状態

|-----------|----------|---------|------|

## 10. SO-05F ETS フォーマット詳細解析（2026-06-27）

### 10-1. ETS ファイル構造（共通）

SO-05F の StationTV! (富士ソフト) が生成する `.ETS` ファイルは、
ワンセグ・フルセグ共通で **標準 M2TS コンテナ**。

```
ETS = 192B × N M2TS パケット（オフセット 0 から）
  各 M2TS パケット = [4B ARIBタイムスタンプ] + [188B TS パケット]
  ※ファイル先頭から直接 M2TS（4B ヘッダなし）
```

ファイルサイズ確認（MPG001〜008）:

| ファイル | サイズ | パケット数 | 分類 |
|---------|-------|-----------|-----|
| MPG001 | 2.2 MB | 11,712 | ワンセグ (~44秒?) |
| MPG002 | 3.1 MB | 16,000 | ワンセグ |
| MPG003 | 14.7 MB | 76,800 | フルセグ |
| MPG004 | 60.9 MB | 317,440 | フルセグ |
| MPG005 | 44.2 MB | 230,400 | フルセグ |
| MPG006 | 20.6 MB | 107,520 | フルセグ |
| MPG007 | 0 B | 0 | 空 |
| MPG008 | 336.2 MB | 1,751,040 | フルセグ（最大）|

### 10-2. PID 構成の概観

全ファイルに共通する **Clear（TSC=0）PID**:
- `0x0000` = PAT（フルセグのみ）
- `0x001F` = SIT / Selection Information Table
- `0x1FC8` = **ETS 固有の DRM メタ PMT**（全ファイル共通）

ワンセグ固有スクランブル PID: `0x10`, `0x11`, `0x14`, `0x24`, `0x27`  
フルセグ固有スクランブル PID: `0x111`, `0x112`, `0x0C18`, `0x0012`(EIT)

### 10-3. PID 0x1FC8 の役割（ETS 固有 DRM メタ PMT）

PID `0x1FC8` は全録画に1パケットのみ存在し、**必ず TSC=0（クリア）**。  
このパケットは MPEG-2 PMT (table_id=0x02) を保持し、**DRM 管理情報**を格納する。

**PAT でのマッピング**:
- プログラム 3480 (0x0D98) → PMT PID 0x1FC8

### 10-4. ワンセグ PMT（CA なし）

```
PMT for program 0x8580 (ワンセグ):
  PCR_PID: 0x05FF
  program_info_length: 3B
    desc 0xC1 (1B): 0x88  ← SHARP/富士ソフト独自フラグ
  Streams:
    0x0D (private data)  PID=0x0580  ← データ放送
    0x1B (H.264 video)   PID=0x0581  ← ワンセグ映像
    0x0F (MPEG-2 AAC)    PID=0x0583  ← ワンセグ音声
    0x06 (private)       PID=0x0587  ← 字幕
    0x0D (private data)  PID=0x0589-0x058B
```

**ワンセグは CA_descriptor なし** → Marlin DRM 非適用。  
放送スクランブル（Multi2、TSC=3）のみ残存。

### 10-5. フルセグ PMT（Marlin DRM）

```
PMT for program 0x0C18/0x0C19/0x0C28 (フルセグ):
  PCR_PID: 0x0100 または 0x01FF
  program_info_length: 120B
    CA_descriptor (tag=0x09, 118B):
      CA_system_id: 0x4AF4  ← Marlin DRM (Sony SEMC)
      CA_PID:       0x0001  ← ECM 配送 PID
      private_data (114B):  ← Marlin ライセンス情報
  Streams: なし（ストリーム構造はMarlin ライセンス管理）
```

**フルセグは Marlin DRM + Multi2 スクランブルの二重保護**。  
ストリーム PID がPMT に記載されない → DRM解除後のみ構造判明。

### 10-6. Marlin DRM ライセンス URN（全フルセグ共通構造）

CA private_data の `114B` は以下の構造:

```
\x02\x00\x01 + ASCII:
  nsemc-import:urn:marlin:organization:semc:licensesign:
  000700010114cb8b:
  <content_id (40桁10進数)>
```

各フィールドの解釈:

| フィールド | 値 | 意味 |
|-----------|-----|-----|
| `nsemc-import` | 固定 | NSEMC（Sony SEMC Marlin）ライセンスインポート |
| `urn:marlin:organization:semc` | 固定 | Sony Ericsson/SEMC 組織 URI |
| `licensesign` | 固定 | ライセンス署名種別 |
| `000700010114cb8b` | **全ファイル共通** | デバイスまたはモデル固有 ID |
| `<content_id>` | **録画ごとに異なる** | コンテンツ固有ライセンス識別子 |

フルセグ各録画の content_id:

| 録画 | content_id (40桁) |
|------|------------------|
| MPG003 | `5392370535524993621672029097206749352495` |
| MPG004 | `9804269605137753104639392562652256649889` |
| MPG005 | `9152970999746559227230041657482925198053` |
| MPG006 | `5312532595902795683290827291008727372019` |
| MPG008 | `7060827625317993780899718766410030825645` |

`000700010114cb8b` はデバイス固有 ID（または SO-05F のモデル固有 ID）と推定。

### 10-7. フルセグ復号に必要なもの

```
SO-05F の Marlin DRM 個人化データ（デバイスバインドライセンス）
    ↓
content_id でライセンスを特定
    ↓
Marlin ライセンスから Multi2 スクランブル鍵 Ks を取得
    ↓
ECM（PID=0x0001）から Ks でコンテンツを復号
    ↓
平文 M2TS
```

`libDxDrmServer.so`（PRIVATE/android_libs/ に取得済み）が Marlin 処理を担当。  
デバイスバインドライセンスは SO-05F の TrustZone / 安全なストレージに保存。

### 10-8. ワンセグ復号に必要なもの

```
ECM（PID: ワンセグ専用 or 共通）に含まれる Multi2 スクランブル鍵 Ks
    ↓
Multi2 でスクランブル解除
    ↓
平文 MPEG-2 TS（ワンセグ品質）
```

Marlin DRM が不要な分、標準 ISDB-T/B-CAS 相当の方法でアクセス可能。  
ECM PID は現状不明（PMT にストリーム情報なし、ワンセグ専用 PMT が別 PID に存在の可能性）。

### 10-9. ETS と SH-06F フルセグの比較

| 項目 | SH-06F フルセグ (.sb1/.FSEG) | SO-05F ETS |
|------|---------------------------|-----------|
| コンテナ | SHARP 独自 M2TS + AES-CBC | 標準 M2TS (192B) |
| ファイル暗号 | AES-128-CBC (device key で secureInfo 復号) | **なし** (ファイルレベル暗号なし) |
| コンテンツ | AES-CBC でブロック暗号化 | Multi2 放送スクランブル残存 |
| DRM | `/system/etc/mccd/sbdb` 由来デバイス鍵 | Marlin DRM (Sony SEMC) |
| 鍵管理 | SHFSEG0001.DB の secureInfo | PMT CA_desc の 114B URN |
| 復号難度 | 高（デバイス固有鍵不明） | 高（Marlin ライセンス必要） |

### 10-10. 追加ファイルの役割

| 拡張子 | 内容 |
|--------|------|
| `.ETS` | 録画コンテンツ本体（192B M2TS、Multi2+Marlin） |
| `.MMA` | 音声メタデータ（28B固定長、フォーマット情報）|
| `.MMO` | 録画メタデータ（1036B、タイムスタンプ等）|
| `.MPI` | 再生リスト情報（296/330B、タイトル情報・タイムコード）|
</content>
