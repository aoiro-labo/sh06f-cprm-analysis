# SH-06F ワンセグ／フルセグ録画データ 解析メモ

最終更新: 2026-06-23

---

## 結論（先に）

| 対象 | 状態 | 理由 |
|------|------|------|
| **ワンセグ** (SD_VIDEO/*.sb1) | **画面キャプチャで救出済み** | XP VM + SD-MobileImpact + バイナリパッチで再生成功 |
| **フルセグ** (PRIVATE/SHARP/FSEG/) | **未解決** | SHARP 独自形式・デバイス紐付き・SD-MobileImpact 非対応 |
| **番組メタデータ** | **✅ 全件復元済み** | PGI/MOI/SQLite DB から平文で抽出 |

「ワンセグは DRM がない」は誤解である。SH-06F が SD カードへ録画した時点で CPRM による再暗号化が適用される。
実測でもエントロピー ≒ 8.0・TS 同期バイト(0x47)の規則的出現なしを確認しており、全域が暗号化されている。

詳細な再現手順: **[`doc/ワンセグ救出_再現手順.md`](ワンセグ救出_再現手順.md)**  
暗号構造の技術詳細: **[`doc/暗号構造解析.md`](暗号構造解析.md)**

## 1. データ構成

SD カードのユーザー領域（通常のファイルコピーで取得できる範囲）の構造:

```
microSD ルート/
├── SD_VIDEO/                       ← ワンセグ（SD-Video 規格・CPRM 暗号化）
│   ├── MGR_INFO/
│   │   ├── MGR_DATA   (256B  "V1" ヘッダ)
│   │   └── PRG_MGR    (5080B "V2" 番組管理)
│   └── PRG001〜PRG012 / PRG00A〜PRG00F / PRG010〜PRG012/
│       ├── MOV001.sb1  (★ AV ストリーム本体・CPRM 暗号化)
│       ├── MOV001.moi  ("V6" 動画オブジェクト情報・平文)
│       ├── MOV001.mai  ("V8" 音声情報・平文)
│       └── PRGxxx.PGI  ("V3" 番組情報・Shift-JIS 平文)
│
├── PRIVATE/SHARP/FSEG/             ← フルセグ（SHARP 独自形式）
│   ├── 10001/
│   │   ├── 0000100010  (★ AV ストリーム 883MB・独自暗号化)
│   │   ├── 0001100010  (576B 暗号化鍵ブロック)
│   │   └── SHFSEG0001.DB  (SQLite・メタデータ+暗号 BLOB)
│   ├── 10002/
│   │   ├── 0000100020  (★ AV ストリーム 1,254MB)
│   │   ├── 0001100020  (576B 暗号化鍵ブロック)
│   │   └── SHFSEG0001.DB
│   ├── 10003/
│   │   ├── 0000100030  (★ AV ストリーム 1,233MB)
│   │   ├── 0001100030  (576B 暗号化鍵ブロック)
│   │   └── SHFSEG0001.DB
│   └── 10004/
│       ├── 0000100040  (★ AV ストリーム 1,246MB)
│       ├── 0001100040  (576B 暗号化鍵ブロック)
│       └── SHFSEG0001.DB
│
├── 0001100010                      ← 576B 鍵ブロック（FSEG/10001 のルートコピー）
├── 0001100020                      ← 576B 鍵ブロック（FSEG/10002 のルートコピー）
│                                      ※ 0001100030・0001100040 はルートコピー不在
└── SHFSEG0001.DB                  ← DB のルートコピー
```

**重要**: CPRM の保護領域（MKB・Media ID・暗号化タイトル鍵群 SD_BIND/）は
通常のファイルコピーには含まれない。CPRM 認証コマンド (AKE) 経由でのみ読み出せる。

## 2. 各ファイルの解析結果

### 2-1. ワンセグ PGI（番組情報・平文）

`PRGxxx.PGI` はマジック `56 33`（"V3"）の SHARP 独自構造。
offset 0x2F から番組タイトル（Shift-JIS）、その他フィールドに録画日時・チャンネル番号が平文で格納されている。
全 18 番組のメタデータを `doc/_oneseg_titles.txt` に抽出済み。

### 2-2. ワンセグ MOI（"V6"・平文）

録画開始日時（BCD/バイナリ）・総再生時間・チャプタ境界テーブルを格納。映像データは含まない。

### 2-3. ワンセグ MAI（"V8"・平文）

音声情報（サンプリングレート・チャンネル数等）を格納。

### 2-4. フルセグ DB（SQLite・平文メタ ＋ 暗号 BLOB）

```sql
CREATE TABLE fseg(
  contentsId INTEGER UNIQUE,
  title TEXT,                -- 番組タイトル（平文・UTF-8）
  remoteCtrlID INTEGER,
  channelName TEXT,          -- 放送局名（平文）
  recStartDate TEXT,
  recEndDate TEXT,
  playTime INTEGER,
  dataSize INTEGER,
  copyCount BLOB,            -- 64B 暗号化（コピー制御）
  serviceID INTEGER,
  contentsType INTEGER,
  saveDate TEXT,
  secureInfo BLOB,           -- 64B 暗号化（タイトル鍵/権利情報）
  emmNum BLOB,               -- 64B 暗号化（EMM 関連）
  resumeTime INTEGER
);
```

`title`・`channelName` は平文で取得済み（`doc/_fseg_db_dump.txt`）。
`secureInfo`・`emmNum`・`copyCount` は各 64B の暗号化 BLOB で番組ごとに異なる。
これらの復号には SH-06F デバイス内部の秘密鍵が必要であり、外部から取り出せない。

### 2-5. 576B ファイル（`0001100010` 等）

全域高エントロピー（≒ 8.0）で暗号化済み。
録画 1・2 で同一内容（MD5 一致）、録画 3・4 でそれぞれ異なる。
MKB バージョン・カウンタ等のカード状態に依存する鍵ブロックと推定される。Kmu なしには復号不可。

## 3. 暗号化の確認（実測）

| 対象 | サイズ | 全体エントロピー | 先頭 256B | TS 同期(0x47) |
|------|--------|----------------|----------|--------------|
| PRG001/MOV001.sb1 | 444KB | **7.953** | 3.62（構造ヘッダ部） | 規則的出現なし |
| PRG007/MOV001.sb1 | 17.4MB | **7.954** | 3.63 | 同上 |
| FSEG/10001/0000100010 | 883MB | **8.000** | — | — |

エントロピー ≒ 8.0 はランダムデータと等価であり、暗号化を示す。
188/192B 周期での TS 同期バイト出現がない点も平文 TS でないことを裏付ける。

`.sb1` 先頭の `0x04: 47 00 1F 10 ...` という断片は暗号化前ヘッダ部の名残であり、
ストリーム全体が平文 TS であることを意味しない（§2-4 の既存スクリプト `dec.py` はこの前提に基づいており機能しない）。

## 4. CPRM の仕組みと復号不能の理由

### 4-1. 鍵階層

```
[デバイス鍵（秘密・ライセンス品）]
    + [MKB（カードのシステム領域・AKE 必須）]
    → メディア鍵 Km

Km + [Media ID（カード製造時固有値・AKE 必須）]
    → メディア固有鍵 Kmu

Kmu + [暗号化タイトル鍵 ETK（保護領域 SD_BIND/・AKE 必須）]
    → タイトル鍵 TK (56bit C2 鍵)

TK + C2-CBC → コンテンツ復号
```

MKB・Media ID・ETK のいずれも AKE（認証鍵交換）付き SCSI コマンド経由でしか読み出せない。
`dd`・WSL・通常コピーでは保護領域の内容は得られない。
デバイス鍵は各機器にライセンスされた非公開値であり、外部から取り出せない。
Media ID はカード固有のため、別カードにコピーしても認証が通らない。

### 4-2. 正規経路の検証結果

**Panasonic DIGA（USB 接続）**:

- SD カードを USB 接続した BN-SDCMP3 経由で DIGA に認識させると、
  「持ち出し番組一覧」に 12 番組（2 ページ分・18 本）が表示された（スクリーンショット参照）

![DIGA 持ち出し番組一覧](../%E3%83%87%E3%82%A3%E3%83%BC%E3%82%AC%E3%81%A7%E5%85%83%E3%81%AEmictoSD%E3%82%92%E8%AA%AD%E3%81%BF%E8%BE%BC%E3%82%93%E3%81%A0%E3%81%A8%E3%81%8D%E3%81%AE%E3%82%84%E3%81%A4%E2%91%A0.png)

- 再生を試みると「**持ち出し番組の再生はできません**」のダイアログが表示され、再生できない

![DIGA 再生エラー](../%E3%83%87%E3%82%A3%E3%83%BC%E3%82%AC%E3%81%A7%E5%85%83%E3%81%AEmictoSD%E3%82%92%E8%AA%AD%E3%81%BF%E8%BE%BC%E3%82%93%E3%81%A0%E3%81%A8%E3%81%8D%E3%81%AE%E3%82%84%E3%81%A4%E2%91%A1.png)

DIGA は録画機として動作するため「持ち出し番組（他機での再生専用コンテンツ）」を再生するロールを持たない。
加えて USB 接続では CPRM の AKE が通らないという技術的制限もある。
ダビング操作もリーダーをダビング元として認識しないため不可。

**SD-MobileImpact（Windows 11）**:

起動時に `mirssom.dll`（ミュージックソムリエ機能。CPRM とは無関係）の
use-after-unload でアクセス違反 0xC0000005（offset 0x00020096）が発生してクラッシュする。
Windows イベントログ ID 1000・Procmon ログで確認済み。
互換モード（WINXPSP3）・管理者権限の組み合わせを試みたが改善しなかった。

### 4-3. 既存スクリプト `dec.py` の評価

B-CAS カード + Multi2 によるライブ放送デスクランブルを模した実装。本ケースへの適用は不可:
1. SD 録画コンテンツは Multi2（放送スクランブル）ではなく CPRM/C2（録画時再暗号化）
2. タイトル鍵 Kw はモック値で本物は入手不可
3. 中身は平文 TS でないためパケット単位処理の前提が崩れている

## 5. 試みた手段・評価

### 機能しない手段

- `.sb1` を `.ts`/`.m2ts` にリネームしての再生（暗号化済みのため不可）
- WSL / Linux / ffmpeg / TS パーサ（鍵がないため復号段階で停止）
- DVD-VR 用 CPRM Decrypter（SD カード CPRM は鍵系統が別・保護領域が必要）
- 56bit 総当たり（2^56 試行が必要・家庭用 PC では非現実的）

### 有効な手段（ワンセグ）

**✅ XP SP3 VM + SD-MobileImpact L032 + バイナリパッチ**

詳細: [`ワンセグ救出_再現手順.md`](ワンセグ救出_再現手順.md)

バイナリパッチにより 86000012（SDHC 拒否エラー）を回避し、
ゲスト OS のハードウェアアクセラレータを無効化して黒画面を解消する。
元 microSD（BN-SDCMP3 経由 USB パススルー）が必要。
再生成功後は OBS Studio でリアルタイム画面・音声キャプチャにより映像を救出する。
