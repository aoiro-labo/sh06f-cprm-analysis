## 0. 結論（先に）

| 対象 | 映像として復号・再生できるか | 理由 |
|------|------------------------------|------|
| **ワンセグ** (`SD_VIDEO/PRGxxx/MOV001.sb1`) | **PCにコピーしたファイルだけでは不可** | CPRM暗号。復号鍵がSDカードの「保護領域」にあり、今回のコピーには含まれていない |
| **フルセグ** (`PRIVATE/SHARP/FSEG/...`) | **不可（さらに困難）** | SHARP独自＋デバイス紐付け暗号。鍵はSH-06F本体のセキュア領域にあり取り出せない |
| **番組メタデータ**（題名・局・日時・長さ） | **✅ 全て救出済み** | 管理ファイル・DBに平文(Shift-JIS)で格納されていた → 本書 6章に一覧 |

要点：**「ワンセグはDRMが無い」というのは誤解**でした。本体で**SDに録画した時点でCPRMで暗号化**されています（放送波そのものに追加暗号が無いのは事実ですが、録画ファイルは別）。実測でも `.sb1`・フルセグ本体ともに**エントロピー≒8.0／TS同期バイト無し＝全域暗号化**を確認しました（3章）。

> 📑 **暗号方式・ファイル構造の詳細リバースエンジニアリングは別紙
> [`doc/暗号構造解析.md`](暗号構造解析.md) に分離**（C2-CBC特定／6144Bユニット構造／
> 56bit鍵の攻撃面など）。鍵が無い前提で「中身の解析」を進めたい場合はそちら。

そして復号に必要な鍵材料（Media ID・MKB・暗号化タイトル鍵）は、SDカードの**通常のファイルコピーでは絶対に読めない隠し保護領域**にあります。**WSL/Linuxを使っても解決しません**（OSの能力の問題ではなく、鍵が物理的にコピーに入っていないため）。

> **唯一の現実的な再生先は SH-06F 本体のテレビアプリ**です。元のmicroSDとSH-06F本体のペアリングが生きていれば本体では再生できます。それ以外（DIGA経由のダビング含む）は4章の通り全て塞がっています。

## 1. データ構成

PCにコピーされているのは、SDカードの**ユーザー領域（通常見えるファイル）だけ**です。

```
ルート/
├ SD_VIDEO/                      ← ワンセグ（SD-Video規格）
│  ├ MGR_INFO/
│  │  ├ MGR_DATA   (256B  "V1"ヘッダ)
│  │  └ PRG_MGR    (5080B "V2"ヘッダ・全番組のタイトル等)
│  └ PRG001〜PRG012/             ← 1番組=1フォルダ（12番組）
│     ├ MOV001.moi  (動画オブジェクト情報 "V6"。日時/チャプタ等・平文)
│     ├ MOV001.mai  (音声情報 "V8"・平文)
│     ├ MOV001.sb1  (★AVストリーム本体・CPRMで暗号化)
│     └ PRGxxx.PGI  (番組情報 "V3"。題名/説明/日時/CH・Shift-JIS平文)
│
├ PRIVATE/SHARP/FSEG/            ← フルセグ（SHARP独自）
│  └ 10001〜10004/               ← 1番組=1フォルダ（4番組）
│     ├ 0000100010 (★AVストリーム本体・暗号化。883MB〜1.25GB)
│     ├ 0001100010 (576B 鍵ブロック?・暗号化)
│     └ SHFSEG0001.DB (16KB SQLite。題名/日時/secureInfo等)
│
├ 0001100010 / 0001100020        ← 上記576Bファイルのコピー(同一)
├ SHFSEG0001.DB                  ← 10001のDBのコピー(同一)
└ (各種 .lst = 過去の解析ダンプ)
```

> ユーザー談：元のSDカードでは `SD_VIDEO` と `PRIVATE` が他のルートフォルダとともに存在。
> → しかし**CPRMの保護領域（SD_BIND 等）はエクスプローラのコピーでは出てこない**ため、
> 　「全フォルダをコピー」しても鍵は得られていない（4章）。

## 2. 各ファイルの中身（判明分）

### 2-1. ワンセグ PGI（番組情報・平文）
`PRGxxx.PGI` は先頭 `56 33`("V3") のSHARP独自構造。**題名・説明・録画日時・チャンネルが Shift-JIS 平文**で入っている。オフセット `0x2F` から題名文字列。
→ 全12番組の題名を抽出済み（6章 / `doc/_oneseg_titles.txt`）。

例：`PRG001.PGI` に `2025/12/06 21:27:01.000`〜`21:27:14.000`、CH "680" "03" 等。

### 2-2. ワンセグ MOI（"V6"・平文）
録画開始日時(BCD/バイナリ)、総再生時間、チャプタ境界テーブル等。映像は含まない。

### 2-3. フルセグ DB（SQLite・平文メタ＋暗号BLOB）
テーブル定義：
```sql
CREATE TABLE fseg(
  contentsId INTEGER UNIQUE, title TEXT, remoteCtrlID INTEGER,
  channelName TEXT, recStartDate TEXT, recEndDate TEXT,
  playTime INTEGER, dataSize INTEGER,
  copyCount BLOB,            -- 64B 暗号化（コピー制御）
  serviceID INTEGER, contentsType INTEGER, saveDate TEXT,
  secureInfo BLOB,           -- 64B 暗号化（タイトル鍵/権利情報と推定）
  emmNum BLOB,               -- 64B 暗号化（EMM関連と推定）
  resumeTime INTEGER);
```
- `title`/`channelName` … Shift-JIS 平文 → 救出済み（6章）。
- `secureInfo`/`emmNum`/`copyCount` … 各64Bの暗号化BLOB。番組ごとに全て異なる。
  これらが鍵・権利情報だが、**復号にはSH-06F本体内の秘密鍵が必要**で取り出せない。

### 2-4. 576Bファイル（`0001100010` 等）
全域高エントロピー＝暗号化。録画1・2では同一、録画3・4では各々異なる（MD5一致を確認）。
→ カード状態（MKBバージョン/カウンタ等）に依存する鍵ブロックと推定。単体では復号不能。

## 3. 暗号化の確認（実測）

`.sb1`・フルセグ本体ともに**先頭に小さな平文ヘッダ、その後ろは全域が高エントロピー**。

| 対象 | サイズ | エントロピー(全体) | 先頭256B | TS同期(0x47) |
|------|--------|--------------------|----------|--------------|
| `PRG001/MOV001.sb1` | 444KB | **7.953** | 3.62(構造ヘッダ) | 188/192両方とも無し |
| `PRG007/MOV001.sb1` | 17.4MB | **7.954** | 3.63 | 規則的同期無し |
| `FSEG/10001/0000100010` | 883MB | **8.000** | — | — |

- エントロピー≒8.0 = ランダム同然 → **暗号化されている**。
- 188/192バイト周期で `0x47`(MPEG-2 TS同期バイト)が出現しない → **平文TSではない**。
  → `dec.py` が前提にした「中身は素のTSで、Multi2の鍵さえあれば解ける」は成り立たない。

`.sb1` 先頭の `00000004: 47 00 1F 10 ...` のような断片は、暗号化前のヘッダ／パケット境界の名残で、
ストリーム全体が平文TSであることを意味しない。

## 4. なぜ復号できないか（CPRMの仕組み）と正規経路の検証

### 4-1. CPRMの鍵階層
```
[デバイス鍵(秘密・ライセンス品)] + [MKB(カードのシステム領域)] → メディア鍵 Km
Km + [Media ID(カード製造時の読取専用領域)]                    → メディア固有鍵 Kmu
Kmu で [暗号化タイトル鍵(保護領域 SD_BIND)] を復号           → タイトル鍵
タイトル鍵 + C2暗号 で本体(.sb1 / フルセグ)を復号
```
復号に最低限必要なのは **Media ID・MKB・暗号化タイトル鍵** の3点。これらは全て
**SDカードの「保護領域／読取専用領域」**にあり、

- **通常のファイルコピーでは取得不可**（CPRM認証コマンド AKE 経由でしか読めない）。
  → 今回PCにあるのはユーザー領域のみ。**鍵が1つも入っていない。**
- 仮に取り出せても**デバイス鍵は秘密**（DVD-VRと違いSDカード用の有効な鍵は出回っていない）。
- Media IDはカード固有のハードウェア値で、別カードへコピーしても無効。

→ **これは設計通りの「コピー不可」であり、コピー済みファイルからの復号は原理的に詰み。**
　WSL/Linux/専用ツールを使っても、無い鍵は作れないので状況は変わらない。

### 4-2. 正規経路（本体・DIGA）の検証結果
- **DIGAに同じパナ製SDリーダーをUSB接続**：USB機器として認識され、持ち出し番組一覧に
  12番組が表示される（スクショ①）。**しかし**
  - 再生 → 「**持ち出し番組の再生はできません**」（スクショ②）
  - ダビング → リーダーが**ダビング元デバイスとして認識されず不可**（ユーザー確認）
  → 正規のPanasonic機ですら弾く＝**録画機(SH-06F)に紐付いた持ち出し専用コンテンツ**。
- フルセグも同様にデバイス紐付け。

### 4-3. `dec.py`（既存スクリプト）の評価
B-CASカード＋Multi2でのライブ放送デスクランブルを模した実装だが、本ケースには**不適合**：
1. SD録画コンテンツはMulti2(放送スクランブル)ではなく**CPRM/C2(録画時の再暗号化)**。
2. 鍵`Kw`は固定モック値で、本物のタイトル鍵は前述の通り入手不能。
3. そもそも中身は平文TSではない（3章）ので、TSパケット単位の処理が前提から崩れる。
→ このアプローチでは復号できない。アーカイブとして残すのは可だが、成功の見込みは無い。

## 5. 試して無駄なこと / 唯一の望み

### 効果が無い（やらなくてよい）
- ❌ `.sb1` を `.ts`/`.m2ts` にリネームしての再生（暗号化済みなので不可）
- ❌ WSL/Linux・ffmpeg・TSパーサ類（鍵が無いので復号段階で必ず止まる）
- ❌ DVD-VR用CPRM Decrypter等の流用（SDカードCPRMは鍵系統が別、かつ保護領域が必要）
- ❌ コピー済みファイルだけを相手にした総当たり（128bit級鍵で非現実的）

### 望みのある再生ルート（いずれも要・元microSD）
- ✅ **SH-06F本体のテレビアプリで再生**：元microSD＋本体のペアが生きていれば本体では見られる。
  画面録画（HDMIキャプチャ等）が事実上の救出手段になる場合がある。
  ※ アナログ出力制限がかかる可能性あり。要実機確認。
- ✅ **【有力】旧世代パナソニック ゴリラ/ストラーダポケットで再生（ワンセグのみ）**
  - 2010年前後の機種（例 CN-MW200D / CN-MW250D / ストラーダポケット CN-MP系 等）は
    **「SDワンセグ動画」= SD VIDEO規格(ISDB-T Mobile Video Profile, H.264)** の再生に対応。
    取説に「携帯電話などで録画したワンセグもSD VIDEO規格で保存したものは再生可」と明記。
  - **CPRMの「番組持ち出し」コンテンツを再生する設計**なので、SH-06Fの`SD_VIDEO`が読める見込み。
  - **DIGAで弾かれたのにゴリラなら可能性がある理由**：DIGAは録画/ダビング機なので
    他機が作った「持ち出し番組」を**再生しない（作る側で読まない）**。一方ゴリラは
    SDワンセグ動画の**再生専用デバイス**として動くため役割が違う。
  - 必須条件：①**元microSD（保護領域が無傷）** ②**本体のSD/microSDスロットに直挿し**
    （microSD→SD変換アダプタ。**USBリーダー経由は不可**＝DIGA失敗の一因）
    ③**対応する旧世代機**（新しい CN-G系は著作権保護SD動画/番組持ち出し非対応、素のMP4のみ）。
  - 注意：**フルセグ(FSEG)は再生不可**（カーナビはSHARP独自形式を読まない）。ワンセグ限定。
  - 救出したい場合：ナビのAV出力→キャプチャ（アナログ）も理論上可。ただし
    コピー制御信号(CGMS-A)で録画阻止/劣化の可能性あり。要実機確認。
- ⚠️ **ハード破損注意**：SDカード型リーダー（厚み・形状が変則）を機器のSDスロットへ無理に
  挿すと**スロットのバネ機構が固着・破損する恐れ**（DIGAで発生しかけた実例あり）。
  必ず**正規厚のmicroSD→SD変換アダプタ**を使い、無理な力をかけない。
- △ SD-MobileImpact：CPRM対応で**本来は本体扱いで再生/移動できた**が、
  追加DLサーバ停止で本ソフトの機能が不全。仮に動いても元カードが必要。
